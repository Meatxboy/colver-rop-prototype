# Conventions

How code in this repo is structured. Match the existing patterns; deviate only with reason.

## File and module layout

- One file per page or major feature in `design/`. Files >800 lines are a soft ceiling — split by adding a new file and inserting it into the script order in `design/index.html`.
- Every `.jsx` / `.js` file ends with `Object.assign(window, { Symbol1, Symbol2, … })`. New components must be added there to be reachable from later files.
- **No imports / requires / exports.** The project has no module system. All inter-file references go through `window`.

## React idioms

- **Functional components only.** State via `useState`, derived values via `useMemo`, effects via `useEffect`, refs via `useRef`.
- Hooks are destructured **once** in `ui.jsx` (`const { useState, useEffect, useRef, useMemo, useCallback, Fragment } = React;`) and treated as globals afterwards. Don't redeclare them.
- Use `Fragment` (also globally available) instead of `<>` when you need a key.
- Don't write `import React`. `React` is a UMD global.
- `ReactDOM.createRoot(...)` is called exactly once, in `app.jsx`. Don't add another root.
- Prefer `useState(() => initFn())` for expensive initial values (existing pattern: `useState(() => initTasks(window.MOCK_DATA))` in `app.jsx:14`).

## Naming

- **Components:** `PascalCase`. File names are kebab-case (`ai-panel.jsx`) but export `PascalCase` symbols (`AiPanel`).
- **Helpers / utilities:** `camelCase` (`formatTime`, `scoreToColor`, `initTasks`, `initNotifications`, `cn`).
- **Constants & module-level data tables:** `UPPER_SNAKE` (`EMPLOYEES`, `KANBAN_COLS`, `TASK_PRIO`, `NEXT_STEPS`, `CALL_RESULTS`).
- **CSS classes:** kebab-case. Two flavours coexist:
  - **Semantic component classes:** `card`, `card-header`, `tabs-list`, `prio-chip`, `cd-criterion-bar`, `ai-msg-user` — the primary system.
  - **Shadcn-flavoured utility-like classes:** `bg-primary`, `bg-secondary`, `bg-success-soft`, `text-primary-foreground` — defined as bespoke CSS rules in `styles.css`, **not** generated by Tailwind. Use them for variants; don't expect generic Tailwind utilities to work.

## Styling

- Prefer existing CSS classes from `design/styles.css`. Reach for inline `style={{ … }}` for one-off layout tweaks (spacing, color from a CSS var). Inline styles are common in this codebase and accepted, but extract a class once a style repeats more than ~3 times.
- Use CSS custom properties (`var(--primary)`, `var(--border)`, `var(--muted-foreground)`) instead of literal hex when you can — they support the theming layer.
- Score thresholds (10-pt normalized): `≥8` good, `≥6` warn, `≥4` default, `<4` bad. The `ScoreCell` primitive normalizes 5-pt to 10-pt internally (`max=5` doubles the value before bucketing).
- **Z-index ladder.** Use these tiers; don't invent intermediate values:
  | Tier | Element |
  |---|---|
  | 10  | AI panel |
  | 15  | Topbar |
  | 20  | Sidebar |
  | 50  | Tweaks panel |
  | 110/120 | Notifications drawer (backdrop / drawer) |
  | 200 | ManagerModal |
  | 300 | CallModal |
  | 400 | TaskCreateModal / TaskDetailModal |
  | 700 | Toast |

## Data flow

- Component code reads from the `data` prop passed down by `<App>` (which owns `window.MOCK_DATA` in `useState`). **Never** read from `window.COLVER_DATA` directly.
- New mock data: add to `data.js` (raw, in the IIFE), then map in `data-adapter.js` to the shape components want. Don't bypass the adapter.
- Mutating actions (e.g., `handleProcess`, `setTasks`, `setNotifications`) live in `app.jsx` and are passed down as callbacks. Pages don't own data state — they receive `data` and call props like `onProcess`, `onCreateTask`, `onOpenCall`.

## Icons

- New icons go in the `Icon` map in `ui.jsx`. Use the lucide-style template:
  - `viewBox="0 0 24 24"`
  - `fill="none"` + `stroke="currentColor"` (use `fill="currentColor"` for filled glyphs only)
  - `strokeWidth="2"`, `strokeLinecap="round"`, `strokeLinejoin="round"`
- Default size 13–16 px. Pass `size={n}` for variation.
- Don't reach for an external icon package. Don't inline ad-hoc SVG inside a component.

## Buttons

- Use `<Button variant="default|secondary|outline|ghost|destructive|success" size="sm|md|lg">`. Custom buttons are tolerated but should adopt the same heights/paddings (h-7=28, h-8=32, h-9=36).
- Disabled state: `disabled` prop + an `opacity:.5; pointer-events:none` style is the existing convention for "looks disabled but parent prevents click."

## Modals

- For full-page-overlay modals (CallModal, TaskCreateModal, TaskDetailModal, ManagerModal) the project uses **inline custom modals**, not the `Modal` component from `ui.jsx`. The custom approach gives full control over headers/footers/scroll/sticky bars.
  - Pattern: a `position:fixed; inset:0; z-index:N; display:flex` outer, an inner `<div onClick={e => e.stopPropagation()}>`, click-outside-to-close on the outer.
  - Lock body scroll: `useEffect(() => { document.body.style.overflow = 'hidden'; return () => { document.body.style.overflow = ''; }; }, [])`.
  - Z-index by tier (see ladder above).
- The lightweight `<Modal>` in `ui.jsx` is fine for small confirms / placeholders, but no large feature in the codebase currently uses it.

## State patterns

- App-level state lives in `app.jsx`. Pages and modals receive callbacks; they don't reach back into `<App>` via context or globals.
- `setData(d => ({ ...d, queue: d.queue.filter(…) }))` — the existing reducer-flavoured updater pattern. Prefer the function form for any state derived from previous state.
- Sort/filter/page state is local to a list component — don't lift it unless multiple consumers need to share.

## Russian text

- All user-facing strings are in Russian. Don't translate without an explicit ask.
- Mixed Russian/English in code comments and identifiers is fine — match the surrounding file. Variable names stay in English (`employee`, `score`, `manager`).

## Mock data conventions

- Date strings: `DD.MM.YYYY HH:mm` for display (`'21.04.2026 15:00'`), `YYYY-MM-DD` for sortable fields (`dueDate`, `createdAt`).
- IDs: `T-NN` for tasks, `C-NNNN` for calls (display) / `'NNNN'` (without prefix in `MOCK_DATA.calls[].id`), `q-NNN` / `m-NNN` / `p-NNN` for queues, `pr-NNN` for processed, `NF1`-`NF6` for notifications.
- Runtime-created entities use `'<prefix>-' + Date.now()` (existing convention in `TaskCreateModal` and `handleProcess`).
- Phone numbers are synthetic and use Russian formatting: `+7 9XX XXX-XX-XX`. Real numbers don't appear in mocks.

## What NOT to add

- Don't add `package.json`, `tsconfig.json`, ESLint, Prettier, Webpack/Vite/Rollup, Jest, or any tooling. The project's value is "open in browser, edit, reload."
- Don't introduce state libraries (Redux, Zustand, Jotai, react-query, SWR). State stays local to `<App>` and its children.
- Don't add network calls (`fetch`, `XMLHttpRequest`, WebSocket). The prototype is offline-only.
- Don't add a router (React Router, etc.). The in-memory `route` object is the convention.
- Don't add CSS-in-JS libraries (styled-components, emotion). Inline `style={{ … }}` and `styles.css` are the two allowed channels.

## When extending the project

- A new page component:
  1. Add the file in `design/`.
  2. Wire it into `design/index.html` in the right script-order slot.
  3. Add a route case in `app.jsx`.
  4. Add an entry to `navItems` in `layout.jsx` if it should appear in the sidebar (and add an icon to the `Icon` map).
  5. Expose it on `window` at the bottom of the file.
- A new mock dataset:
  1. Define raw shape inside the IIFE in `data.js` and add it to the returned object.
  2. Map/transform it in `data-adapter.js` and add to the `MOCK_DATA` object.
  3. Consume `data.<yourField>` in components.
